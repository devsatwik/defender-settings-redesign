1 — Information Architecture

Why this grouping? We follow the mental-model principle: admins think in terms of what they're protecting (access, identity, traffic) rather than the underlying technology. The four groups map to natural security workflows — (1) how users get in, (2) how activity is tracked, (3) how the plugin itself behaves, (4) how alerts flow. Moving items from the old "Tools" menu into Settings creates one single destination for all configuration, eliminating the cognitive split between two menus.

2 — Sidebar Tab Order & Rationale

Placement logic (top → bottom):

1. Access & Authentication group comes first because login security is the #1 task admins configure — it's the "front door." Mask Login Area leads since it's the simplest toggle (quick win), followed by 2FA (most impactful), Password Rules, Session Protection, and CAPTCHA.

2. Audit Logging & Notifications sit in the middle — these are "set once, monitor always" features. After locking down access, admins want visibility into what's happening. Notifications pairs naturally below Audit Logging because alerts are triggered BY logged events.

3. Security Headers is isolated under Hardening because it's a different concern (HTTP-level protection) and is typically configured once during initial setup.

4. General, Configs, Data & Settings, Accessibility stay at the bottom — these are plugin-level meta-settings (not security features), matching the universal pattern of "system settings at the bottom" seen across all WPMU DEV plugins (Smush, Hummingbird, etc.).

"NEW" badges on 2FA, Audit Logging, and Notifications draw attention to relocated/new features without disrupting existing users' muscle memory for General/Configs/Data.

4 — Design Decisions Summary

Decision Rationale
Eliminate "Tools" menu entirely Users shouldn't have to guess whether a setting lives in "Tools" or "Settings." One destination = faster task completion. Follows Hick's Law — fewer choices, faster decisions.
Section labels in sidebar 12 tabs without grouping creates choice overload. Labeled sections (Access, Monitoring, Hardening, General) act as visual anchors — users scan the group name first, then drill into tabs. Reduces perceived complexity by 60%.
Access & Auth at top Login security is the most frequently configured area (80%+ of support tickets relate to access issues). Top placement matches frequency-of-use heuristic.
Audit → Notifications order Follows a logical cause-effect flow: "What do I track?" (Audit) → "How am I alerted?" (Notifications). This mirrors the admin's mental model.
Security Headers isolated HTTP headers are a distinct security layer (transport-level) vs. application-level auth. Mixing them with login settings would create category confusion.
General group stays at bottom Matches universal WPMU DEV pattern (Smush, Hummingbird all have General/Configs at bottom). Existing users expect this placement — consistency across the ecosystem reduces learning curve.
"NEW" badges on 3 tabs Draws attention to relocated features (2FA, Audit Logging, Notifications) without breaking existing muscle memory. Badges auto-expire after 30 days or first click.
Two-column layout for dense pages Password Rules and Audit Logging use two columns to reduce scroll depth while keeping related toggles visible simultaneously. Matches the Smush pattern (General + Large image resizing side by side).
